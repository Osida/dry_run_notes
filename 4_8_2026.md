## 🎯 Lesson Outcomes

### 1. Describe what SQL is
**SQL (Structured Query Language)** is the standard language used to communicate with relational databases.

A **relational database** organizes information into:
- **Rows** → individual records  
- **Columns** → attributes describing each record  

SQL enables storing, retrieving, updating, and managing data.

---

### 2. Explain how SQL Injection Works (Conceptual Overview)
SQL Injection is a vulnerability that occurs when an application incorrectly handles user input, allowing unintended SQL commands to be executed.

Key ideas:
- User input is inserted directly into a SQL query  
- The application does not validate or sanitize the input  
- The database interprets the input as part of the SQL command  

This can lead to unauthorized access or unintended behavior.

---

### 3. Understand the General Process of SQL Injection (High-Level)
In educational environments, students learn how insecure systems behave when input is not validated.

Concepts demonstrated:
- Identifying fields that behave unexpectedly with special characters  
- Observing how SQL syntax affects server responses  
- Understanding how UNION queries reveal structural information  
- Recognizing why secure coding practices prevent these issues  

> These concepts are taught to reinforce secure development practices, not to perform unauthorized actions.

# 🧪 SQL Injection Concepts (Educational Overview)

Below is a conceptual explanation of the steps typically discussed in class.  
These are **not actionable exploitation steps**, but descriptions of how insecure systems behave.

---

## POST Method — Conceptual Flow

### 1. Identifying a Vulnerable Input  
Students observe how certain inputs cause unexpected behavior in insecure applications.

### 2. Determining Query Structure  
Instructors demonstrate how insecure systems reveal:
- Number of columns  
- How mismatched structures behave  

This highlights the importance of parameterized queries.

### 3. Understanding UNION-Based Enumeration  
Students explore how UNION queries work and why insecure systems may unintentionally expose:
- Database names  
- Table names  
- Column names  

This reinforces the need for least‑privilege accounts and proper access controls.

### 4. Querying Data (Conceptual Only)  
Students see how insecure systems may return unintended data when input is not validated.

This emphasizes:
- Never trusting user input  
- Protecting sensitive data  
- Using secure coding practices  

### SQL Version Information  
Students may observe how insecure systems reveal metadata such as:
- Database version  
- Current database name  

This demonstrates the importance of preventing information disclosure.

---

## GET Method — Conceptual Flow

The GET method follows the same conceptual steps as POST, except input is placed directly in the URL.

Students learn:
- How query parameters work  
- How insecure URL parameters affect backend SQL queries  
- Why all incoming data must be sanitized
